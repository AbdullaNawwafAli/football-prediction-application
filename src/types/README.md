# `types/`

Global TypeScript types, interfaces, and enums shared across the entire codebase.

See `src/docs/architecture.md` for how types relate to features and Supabase.

## What belongs here

- Shared domain types: `Match`, `UserProfile`, `LeaderboardRow`
- API shapes used by multiple `services/` modules
- `database.types.ts` — generated by `supabase gen types` (do not hand-edit)
- Generic utility types used in multiple places

## What does NOT belong here

- Types used only within one feature — `features/<name>/types/`
- Component prop types — co-locate in the component file
- Ambient module augmentation — root-level `*.d.ts` files

## Structure

```
types/
├── index.ts                  # Re-exports for importing from #/types
├── database.types.ts         # Auto-generated — supabase gen types
├── match.types.ts
├── prediction.types.ts
└── api.types.ts              # Shared envelopes (e.g. PaginatedResponse<T>)
```

## Example

```ts
// types/match.types.ts
export type MatchStatus = "scheduled" | "live" | "finished" | "postponed";

export interface Match {
  id: string;
  homeTeam: string;
  awayTeam: string;
  kickoffAt: string;
  status: MatchStatus;
  homeScore: number | null;
  awayScore: number | null;
}
```

```ts
// types/api.types.ts
export interface PaginatedResponse<T> {
  data: T[];
  total: number;
  page: number;
  pageSize: number;
}
```

## Conventions

- Use `interface` for object shapes, `type` for unions, intersections, and aliases.
- Name files by domain: `match.types.ts`, not `interfaces.ts` or `models.ts`.
- Prefer importing from `#/types` via `index.ts` rather than deep paths.
- Regenerate `database.types.ts` after schema migrations; never patch generated types by hand.
